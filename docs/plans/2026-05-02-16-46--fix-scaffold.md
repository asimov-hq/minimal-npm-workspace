# Fix Scaffold Plan

## Goal
Make the minimal workspace scaffold internally consistent so that its default verification and development scripts work as advertised.

## Problems To Fix
1. `npm run lint` reports `@typescript-eslint/no-unsafe-call` on obviously safe calls because typed linting cannot reliably resolve `@asimov/minimal-shared` across the current workspace TypeScript setup.
2. `npm run typecheck` fails at the root because the project-reference graph mixes build-oriented and no-emit configs in a way that TypeScript does not accept.
3. `npm run typecheck:packages` fails in `cli`, `server`, and `web` because their build/typecheck configs clear `paths`, which breaks resolution of `@asimov/minimal-shared`.
4. The root workspace scripts use `npm run -ws`, which already emits warnings and should be replaced with current npm flag syntax.
5. The lint config includes some template leftovers and should be tightened so the scaffold is clearer and less surprising.

## Constraints
- Keep the scaffold minimal.
- Preserve the current package split: `shared`, `server`, `cli`, `web`.
- Avoid adding unnecessary dependencies.
- Prefer one coherent TypeScript strategy for dev, lint, build, and typecheck instead of package-specific exceptions.

## Plan
1. Choose and document a single workspace TypeScript resolution strategy.
   - Decide whether internal package imports should resolve through project references plus package metadata, or through source-level `paths` aliases.
   - Use the same strategy consistently across `dev`, `build`, `typecheck`, and `eslint`.

2. Repair the TypeScript project graph.
   - Ensure `shared` is referenced in a way that allows downstream packages to typecheck cleanly.
   - Separate “build config” concerns from “editor/typecheck config” concerns where needed.
   - Make the root `tsconfig.json` and `tsconfig.dev.json` reflect the intended workflow instead of relying on incompatible combinations.

3. Fix package-level type resolution.
   - Update `cli`, `server`, and `web` configs so `@asimov/minimal-shared` resolves during both local development and verification.
   - Remove config mismatches where `paths` exists in one mode but is deleted in another unless there is a clear, necessary reason.

4. Make linting trustworthy.
   - Keep type-aware linting, but only after the TS project configuration is stable.
   - Verify which files should be linted and keep that set intentional.
   - Remove stale ignore entries that do not belong in this scaffold, or justify them clearly.

5. Clean up workspace scripts.
   - Replace deprecated or warning-producing npm flag usage.
   - Ensure `lint`, `typecheck`, `build`, and package-level scripts all run successfully from a clean install.

6. Verify end to end.
   - Run `npm run lint`.
   - Run `npm run typecheck`.
   - Run `npm run build`.
   - Optionally run one dev entrypoint to confirm the scaffold still starts cleanly.

## Expected Outcome
- `npm install` on a fresh clone is sufficient to make the scaffold usable.
- `npm run lint` passes without false-positive unsafe-call errors.
- `npm run typecheck` passes at the root.
- `npm run build` succeeds for all packages.
- The scaffold remains small and understandable.
